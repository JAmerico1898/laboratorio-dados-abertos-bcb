Laboratório de Dados Públicos

Documento técnico do processo de criação e funcionamento da aplicação — uma plataforma web que transforma dados públicos setoriais brutos em visualizações interativas, atualizada de forma autônoma e servida sem banco de dados. Escrito também para servir de framework de replicação: o núcleo genérico está deliberadamente separado da camada de domínio, de modo que outra fonte de dados pública possa reaproveitar toda a espinha dorsal.

01Visão geral

O Laboratório de Dados Públicos é uma aplicação web que consome as APIs abertas de um órgão regulador, congela os dados relevantes em arquivos colunares e os apresenta em módulos temáticos de análise: composição patrimonial, resultado (DRE), carteiras de crédito, taxas de juros, índices financeiros derivados e distribuição regional. O usuário final navega por um hub de cartões, entra em cada módulo, escolhe variáveis e filtros, e recebe gráficos e tabelas de ranking gerados sob medida.

Três decisões definem o produto e explicam quase toda a arquitetura:

🧊

Dados congelados

A aplicação nunca chama a API do órgão em tempo de requisição. Um robô mensal baixa tudo e grava arquivos .parquet versionados no próprio repositório.

🗄️

Sem banco de dados

Não há Postgres, nem ORM, nem migrations. A "base" é o diretório /data. Isso zera custo operacional e torna cada deploy uma fotografia imutável e reproduzível.

🧩

Módulos como padrão

Cada módulo é uma instância de um punhado de padrões (treemap, ranking, índice, mapa). Adicionar análise = declarar dados em constantes, não escrever nova infraestrutura.

O resultado é uma aplicação de custo marginal próximo de zero, que se auto-atualiza quando o órgão publica novos números, e que qualquer pessoa consegue clonar, rodar e auditar por completo, porque todo o "estado do mundo" está commitado no Git.

02Filosofia arquitetural

A aplicação nasceu de uma migração: existia uma versão anterior em Python/Streamlit e foi reescrita em Next.js. Essa herança deixou marcas úteis — a lógica de negócio (classificação, extração, anualização, filtros de materialidade) foi portada função a função do Python para o TypeScript, o que produziu uma camada de dados pura e testável, desacoplada da UI.

Os princípios que sustentam o sistema:

  • Pré-computação sobre computação em tempo real. Todo trabalho caro e lento (chamar a API do órgão, paginar, tratar timeouts) acontece offline, uma vez por mês, num robô. A aplicação só lê arquivos locais.
  • Estado do mundo no repositório. Os dados são artefatos versionados. Um deploy é determinístico: o mesmo commit sempre produz a mesma aplicação com os mesmos números.
  • Domínio isolado em constantes. Praticamente toda a especificidade da fonte de dados (quais relatórios, quais colunas, quais fórmulas, quais rótulos) vive em lib/constants.ts. O código de infraestrutura não conhece o domínio.
  • Servidor fino, cálculo próximo do dado. As route handlers leem parquet, aplicam a regra de negócio e devolvem JSON já pronto para plotar. O cliente não faz cálculo pesado.
  • Estado da UI na URL. Variável selecionada e segmentos ficam nos query params, tornando cada visão compartilhável e navegável pelo botão "voltar".
Por que isto importa para replicação

A separação NÚCLEO / DOMÍNIO é intencional e explícita no código. Trocar a fonte de dados significa reescrever a camada de domínio (o robô de ingestão e o arquivo de constantes) mantendo o núcleo intacto: leitor de parquet, cache, contratos de API, hooks de dados, componentes de visualização e o design system.

05Camada de ingestão DOMÍNIO

O robô scripts/prefetch_data.py é o único ponto que fala com a fonte de dados. É um script Python idempotente, executado pelo GitHub Actions, que produz o conteúdo de /data. Seu fluxo (função main()):

  1. Descobrir o período mais recente. find_latest_quarter_raw() testa, do trimestre atual para trás, qual já foi publicado — porque o órgão publica com 2–3 meses de atraso e de forma irregular. Grava o resultado em latest_quarter.txt.
  2. Baixar o cadastro das instituições daquele período.
  3. Baixar cada relatório do período mais recente (Resumo, Ativo, Passivo, DRE, Crédito por região, Crédito PF, Crédito PJ) e salvar como valores_<periodo>_t1_r<n>.parquet.
  4. Baixar trimestres anteriores apenas dos relatórios que precisam de anualização (Resumo e DRE), somando 4 trimestres — ver §9.3.
  5. Baixar as séries de taxas de juros, uma modalidade por arquivo (taxas_d_<slug>.parquet diárias, taxas_m_<slug>.parquet mensais).
  6. Escrever o manifest.json com período, lista de trimestres, timestamp e inventário de arquivos.

Robustez de ingestão

Como a API do órgão é lenta e intermitente, o robô embute defesas que valem para qualquer fonte pública: max_retries com back-off progressivo (time.sleep(5*(n+1))), time.sleep() entre chamadas para não sobrecarregar o endpoint, e tolerância a arquivos vazios (loga aviso e segue). Os erros do robô nunca chegam ao usuário: se um arquivo falha, a aplicação simplesmente serve o dado anterior já commitado.

Ponto de acoplamento ao domínio

Três coisas neste script são específicas da fonte: (1) a biblioteca cliente do órgão; (2) a lista de relatórios/modalidades a baixar; (3) o formato de período (aqui, trimestres AAAAMM). Para outro setor, é este arquivo — e só ele no back-end — que precisa ser reescrito.

06Camada de dados NÚCLEO

Toda a leitura e o processamento vivem em lib/data.ts. É código de servidor puro (usa fs), sem nenhuma dependência de React. Ele lê os Parquet, aplica as regras de negócio e devolve objetos tipados.

6.1 · Modelo de domínio

O formato dos dados é o clássico "tabela longa" (long/tidy): cada arquivo de valores tem uma linha por (instituição × conta), não uma coluna por métrica. Isso é o que permite ao mesmo código de extração servir dezenas de variáveis diferentes — a métrica é um valor de filtro (NomeColuna), não uma coluna de esquema. As interfaces centrais (lib/types.ts):

interface CadastroRow  { CodInst; NomeInstituicao; Tcb; Sr; ... }
interface InstitutionBase { CodInst; NomeInstituicao; NomeDisplay; NomeReduzido; Segmento }
interface InstitutionRow  extends InstitutionBase { Saldo: number }

Classificação de segmento DOMÍNIO

A função classifySegment() encapsula uma regra regulatória: um campo (Tcb) tem prioridade e, se pertencer a um conjunto de sobreposição, vence; senão cai para outro campo (Sr); senão a entidade é descartada como "Outros". É o exemplo canônico de regra de domínio isolada numa função de 5 linhas — trocável sem tocar em mais nada.

export function classifySegment(row) {
  const tcb = (row.Tcb ?? "").toString().trim().toUpperCase();
  if (TCB_OVERRIDE.has(tcb)) return tcb;      // N1,N2,N4 vencem
  const sr = (row.Sr ?? "").toString().trim().toUpperCase();
  if (VALID_SR.has(sr)) return sr;          // senão S1..S4
  return "Outros";                          // senão descarta
}

Construção da tabela de entidades

buildInstitutionTable(anomes) lê o cadastro, classifica os segmentos, mantém apenas a visão consolidada ("PRUDENCIAL", para não contar a mesma entidade duas vezes), deduplica por código, gera nomes de exibição/curtos e cacheia o resultado. É a "dimensão" que todas as consultas de fato cruzam com os "fatos".

6.2 · Cache e extração

Leitura + cache

readParquet() é o coração do núcleo. Lê o arquivo com hyparquet (import dinâmico, pois é ESM-only), converte BigInt→Number e memoiza o array de linhas num LRUCache de módulo. Como as funções serverless da Vercel são reaproveitadas entre requisições ("warm"), esse cache elimina releituras de disco na maioria dos acessos.

const cache = new LRUCache({ max: 50 });

export async function readParquet(filename, columns?) {
  const key = `parquet:${filename}:${columns ?? "all"}`;
  if (cache.has(key)) return cache.get(key);
  const { parquetReadObjects } = await import("hyparquet");
  const rows = await parquetReadObjects({ file, columns });
  cache.set(key, rows); return rows;
}

As quatro funções de extração NÚCLEO

Toda análise da aplicação se reduz a uma destas quatro operações sobre a tabela longa:

FunçãoO que fazUsada por
extractVariable()Filtra por NomeColuna, cruza com entidades, devolve saldo por instituição.Patrimônio, crédito por região, componentes de índice.
extractVariableAnnualized()Chama extractVariable em 4 períodos e soma por entidade.DRE (fluxos acumulados de 12 meses).
extractCreditVariable()Filtra por Grupo + NomeColuna="Total" (ou por total explícito).Carteiras de crédito PF/PJ.
applyMaterialityFilter()Mantém só entidades acima de limiares de porte (ativo ≥ R$100M e PL ≥ R$20M).Todos os módulos, antes de plotar.
Detalhe de resiliência aprendido em produção

Os nomes de conta da DRE trazem um sufixo de código entre parênteses após um \n (ex.: "Lucro Líquido \n(j) = (g) + (h) + (i)") que o órgão revisa de tempos em tempos. Por isso a extração casa apenas o prefixo legível (s.split("\n")[0].trim()), ignorando o sufixo volátil. É a diferença entre um módulo que quebra silenciosamente a cada revisão do plano de contas e um que sobrevive a ela.

Fechando o núcleo de dados: filterBySegments(), searchBanks() e computeSummary() (total do sistema, contagem e concentração do Top 5) — todos operando sobre InstitutionRow[], sem I/O.

07Camada de API NÚCLEO

As route handlers do App Router são o backend. Todas declaram export const runtime = "nodejs" (precisam de fs), orquestram as funções de lib/data.ts e devolvem JSON já no formato que o gráfico consome. Todas anexam Cache-Control com stale-while-revalidate, então a CDN da Vercel serve a maioria dos acessos sem nem acordar a função.

Endpoint genérico de relatório — /api/ifdata/[report]

Um único handler serve cinco módulos, roteando pelo número do relatório e pelo tipo de extração. É a peça que melhor mostra a economia da arquitetura:

const quarter = getLatestQuarter();
const institutions = await buildInstitutionTable(quarter);

if (report === RELATORIO_CREDITO_PF || report === RELATORIO_CREDITO_PJ)
  data = await extractCreditVariable(quarter, 1, report, varDef.grupo, institutions);
else if (annualized)
  data = await extractVariableAnnualized(getLastNQuarters(quarter,4), report, col, institutions);
else
  data = await extractVariable(quarter, 1, report, col, institutions);

data = await applyMaterialityFilter(data, quarter, institutions);
const { total: systemTotal } = computeSummary(data);   // total do sistema
data = filterBySegments(data, segments);                // só depois filtra
data.sort((a,b) => Math.abs(b.Saldo) - Math.abs(a.Saldo));
return NextResponse.json({ institutions: data, total, systemTotal, count, top5Share, quarter });

Note a ordem: calcula-se o total do sistema inteiro antes de aplicar o filtro de segmentos, para que o percentual mostrado ao usuário tenha um denominador estável.

08Camada de cliente NÚCLEO

Cada módulo é um server component fininho (page.tsx, só metadata) que renderiza um client component (client.tsx) com o estado e os gráficos. A ponte entre a UI e a API é um único hook, useModuleData(), que concentra três responsabilidades:

  • Estado na URL. Variável (?var=) e segmentos (?seg=) são lidos e escritos nos query params via router.replace(..., {scroll:false}) — a visão é compartilhável e o histórico do navegador funciona.
  • Fetch com SWR. Monta a URL da API e usa useSWR com dedupingInterval e revalidateOnFocus:false, evitando refetch redundante.
  • Rótulo de período. Busca /api/quarter (deduplicado por 1h) para exibir o trimestre corrente.
const variable = searchParams.get("var") ?? defaultVariable;
const segments = searchParams.get("seg")?.split(",") ?? DEFAULT_SEGMENTS;
const url = `/api/ifdata/${report}?variable=${enc(variable)}&segments=${segments}${annualized?"&annualized=true":""}`;
const { data, error, isLoading } = useSWR(url, fetcher, { revalidateOnFocus: false });

Cinco dos módulos (todos os de treemap) compartilham ainda o componente components/shared/TreemapModule.tsx, que recebe a definição de variáveis e o número do relatório e monta a página inteira. Um módulo de treemap novo tem, literalmente, dezenas de linhas.

9.1 · Padrão Treemap (5 módulos)

O padrão de maior alavancagem. Cada módulo é uma tupla (relatório, dicionário de variáveis). A página oferece: seletor de variável → filtro de segmentos → métricas-resumo → treemap → tabela Top 20 → busca. Uma variável é declarada assim:

{ key: "ativo_total", nomeColuna: "Ativo Total", label: "Ativo Total",
  icon: "🏦", description: "Total de ativos", relatorio: RELATORIO_RESUMO }

Um VAR_KEY_MAP gerado por um loop no fim de constants.ts mapeia a key curta (que vai na URL) para o nomeColuna real (que filtra o parquet). O handler genérico e o TreemapModule fazem o resto — nenhum código novo por variável.

9.2 · Rankings e séries temporais (Taxas de Juros)

O módulo mais rico em UI, com abas: Rankings (Top 10 maiores e menores taxas por modalidade), Banco Individual (todas as taxas de uma entidade + sua posição no ranking), Gráficos (mediana diária ao longo do tempo) e Download. Cada modalidade é um arquivo Parquet próprio (100–200 mil linhas), endereçado por um slug. Detalhe crítico: a função slugify() do TypeScript precisa produzir exatamente o mesmo slug que a do Python usada na ingestão — é o contrato de nomes de arquivo que liga os dois ciclos.

9.3 · Índices financeiros derivados

O único módulo que calcula em vez de só filtrar. O handler /api/indices lê múltiplos relatórios, anualiza os fluxos (soma de 4 trimestres) e computa razões por entidade. Cada índice é declarado em INDICES com sua fórmula-chave, formato e direção (higherIsBetter). Um switch por índice monta numerador/denominador a partir das funções de extração — exemplo do ROA:

case "roa": {
  const lucro = toMap(await getVarAnnualized("Lucro Líquido \n(j) = (g)+(h)+(i)", RELATORIO_RESULTADO));
  const ativo = toMap(await getVar("Ativo Total", RELATORIO_RESUMO));
  result = computeRatio(lucro, ativo, institutions, segments, 100);
}

Após o cálculo, um filtro estatístico remove outliers além de 3σ da média (números derivados explodem com denominadores minúsculos) e o resultado é ordenado pela direção do índice. O handler também devolve mediana e média para as métricas-resumo.

9.4 · Cartograma (círculos de Dorling)

O único visual que não usa Plotly: é um componente React que desenha SVG puro. Soma o volume de crédito por região (sem filtro de segmento), posiciona um círculo em cada centróide (REGION_CENTROIDS) com área proporcional ao valor (r = min + (max-min)·√(v/vmax)) e anima o crescimento com CSS. É o exemplo de que o padrão "declare os dados, o núcleo desenha" também vale para visualizações totalmente customizadas.

12Deploy & CI/CD NÚCLEO

Automação de dados — .github/workflows/prefetch.yml

Um cron mensal (com retries nos dias 1, 3 e 10, porque a publicação do órgão atrasa) roda o robô Python, e um passo final faz commit de /data apenas se houve mudança (git diff --staged --quiet). O workflow precisa de permissão contents: write.

on:
  schedule:
    - cron: '0 6 1 * *'   # dia 1, 06:00 UTC
    - cron: '0 6 3 * *'   # retry dia 3
    - cron: '0 6 10 * *'  # retry dia 10
  workflow_dispatch: {}       # botão manual

O push que vira deploy

O commit do robô no main dispara o auto-deploy da Vercel. Não há passo de deploy manual no caminho feliz: órgão publica → robô baixa → git commit → Vercel builda → produção atualizada, sem intervenção humana. Detalhe de build: como os Parquet precisam existir no runtime da função, o next.config usa outputFileTracingIncludes para garantir que /data seja empacotado com as route handlers.

13Generalizar o framework para outro setor

A regra de ouro da portabilidade

Se a nova fonte puder ser reduzida ao formato tabela longa — uma linha por (entidade × métrica × período), com uma dimensão de cadastro à parte — então todo o núcleo se aplica sem alteração. O trabalho de portar concentra-se em dois arquivos: o robô de ingestão e o dicionário de constantes. Essa é a promessa do framework.

Roteiro conceitual de adaptação

  1. Mapear as entidades do novo setor (o "cadastro") e sua regra de segmentação.
  2. Mapear os "relatórios" e suas métricas para o formato tabela longa; definir o conceito de período.
  3. Reescrever o robô de ingestão contra a API do novo órgão, preservando o contrato de saída (§5).
  4. Preencher constants.ts: módulos, variáveis, índices, cores, limiares.
  5. Ajustar classifySegment e o switch de índices; rodar a aplicação — os módulos aparecem sozinhos.

15Glossário

Tabela longa
Formato tidy: uma linha por (entidade × métrica), a métrica sendo um valor de filtro e não uma coluna. É o que permite ao mesmo código servir N variáveis.
Anualização
Somar 4 períodos consecutivos de um fluxo (ex.: DRE) para obter o acumulado de 12 meses.
Materialidade
Filtro de porte que descarta entidades irrelevantes antes de plotar.
Visão consolidada
Nível de agregação que evita contar a mesma entidade mais de uma vez ("PRUDENCIAL" na fonte atual).
Slug
Nome normalizado (sem acento, minúsculo) que endereça o arquivo de uma série; deve casar entre ingestão e leitura.
Manifesto
manifest.json: inventário de período, trimestres e arquivos — o índice do lote de dados.
SWR
stale-while-revalidate: serve o cache e revalida em segundo plano — usado tanto na CDN quanto no cliente.
